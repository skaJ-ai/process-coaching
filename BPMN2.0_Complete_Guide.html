Process-Coaching BPMN/L7 가이드북

버전: 2026-02-26 · Single HTML · 내부 SVG 포함

문서 원칙: 1) BPMN 글로벌 표준은 가공 없이 우선 제시, 2) 우리 툴 해석/제약은 후술, 3) 미구현 기능은 제외(병렬 게이트웨이는 가이드 선반영 예외).

1) 프로젝트 목적

본 프로젝트의 목적은 HR 업무를 L7 단위(한 셰이프=한 동작)로 분해해 AS-IS를 명확히 문서화하고, TO-BE 설계까지 이어지는 코칭 체계를 제공하는 것이다.

  • 핵심 질문: 유지/자동화/이관/삭제 가능성을 구조적으로 판단할 수 있는가
  • 산출물: 이해 가능한 프로세스 모델 + 검증 가능한 라벨/구조 품질
  • 운영 원칙: 현업이 직접 작성 가능해야 하며, 규칙 엔진으로 품질을 일관되게 관리
그림 자체가 목적이 아니라, 업무 전환 의사결정을 가능하게 하는 구조화가 목적이다.

2) 설계 근거 (BPMN 표준)

2.1 글로벌 표준 기준

  • 기준 문서: OMG BPMN 2.0.2 (Conformance/Process Modeling)
  • Conformance는 모델링 계열과 실행 계열을 포괄
  • Process Modeling은 Descriptive / Analytic / Common Executable 하위 클래스로 구분

2.2 Descriptive Sub-Class 핵심

분류대표 요소 (OMG Table 2.1 기반)
흐름sequenceFlow, messageFlow
게이트웨이exclusiveGateway, parallelGateway
활동task, userTask, serviceTask, subProcess
이벤트startEvent, endEvent
구조participant, lane, DataObject, TextAnnotation
ISO/IEC 19510:2013은 OMG 2.0.1과 정합성을 명시한다. 실무 적용은 OMG 2.0.2와 교차 확인한다.

2.3 왜 이 표준이 우리 목적에 맞는가

관점적합 이유프로젝트 효과
현업 작성성가시 요소 중심으로 해석이 쉬움작성 진입장벽 감소
품질 관리구조/라벨 규칙화가 용이L7 자동 검증 안정화
확장성필요 시 Analytic/Executable 방향 확장 가능단계적 고도화 가능

3) 우리 툴 모델링 프로파일

우리는 Descriptive 철학을 차용하되, 제품 목적에 맞게 요소를 선별한 실무 프로파일로 운영한다.

정리: 표준 기반(Descriptive) + 제품 맞춤 제한(가독성/검증성 우선) 조합이다.

4) 요소별 작성 가이드

4.1 시작/종료 노드

  • Start는 진입점, End는 완료 조건을 명확히 닫는다.
  • Start 다중 사용은 의도적 구조일 때만 허용, End는 최소 1개 필수.

4.2 프로세스 노드 (Task)

  • 동사 + 목적어로 작성: 예) 결재 요청서를 검토한다
  • 한 셰이프=한 동작 원칙 준수
금지 예: "검토하고 승인한다"(복수 동작), "업무를 처리한다"(모호 동사)

4.3 판단 노드 (Decision / XOR)

  • 조건 분기점에서만 사용
  • 분기 엣지 라벨(Yes/No 또는 조건) 필수
  • 마름모 꼭지점에서 엣지가 나가도록 배치

4.4 병렬 노드 (Parallel / +)

  • + Split/Join 쌍으로 사용
  • Split 후 모든 경로는 Join으로 회수
병렬 노드는 본 문서에서 표기/작성 원칙을 먼저 고정하고, 제품 동작 반영 시점에는 UI/검증 규칙과 함께 동기화한다.
Split만 있고 Join이 없거나, Join 없이 End로 가는 구조는 금지.

4.5 역할 구분선/Phase 구분선

  • 역할 구분선(Y): 누가 수행하는지 표시
  • Phase 구분선(X): 어느 단계에서 수행되는지 표시

5) 빈 캔버스 작성 절차

  1. 범위를 고정하고 Start/End 먼저 배치
  2. 역할 구분선(Y) 여부 결정, 역할 라벨 입력
  3. Phase 구분선(X) 3~5개로 단계 먼저 분리
  4. 화살표 없이 태스크 후보를 먼저 배치 (Top-down)
  5. Happy Path를 Start→End로 연결
  6. Decision/Parallel을 필요한 지점에만 추가
  7. 예외/루프를 추가하되 탈출 경로 확인
  8. L7 규칙 + 구조 규칙으로 최종 검증
작성 순서의 핵심은 "상세보다 구조 먼저"다.

6) L7 품질 기준 (First_Guideline 반영)

아래는 First_Guideline.md 체크리스트를 현재 엔진 기준과 맞춰 통합한 운영 규칙이다.

6.1 원칙

  • 필수 항목 100% 충족을 목표로 작성
  • 문장 단독으로 제3자가 수행 가능해야 함
  • AS-IS 문서화 기준(개선 제안 문구는 분리)

6.2 체크 축 (First Guideline 통합)

축핵심 체크운영 방식
구조·형식단일 행동, 목적어 명확성, 문장 독립성R-05/R-07 중심
수행 맥락위치/화면/채널/입력 대상 명시라벨+메타데이터(systemName/input/output)
판단판단 기준/결과/분기 표현Decision 라벨 + edge 라벨(S-08, R-08)
산출물/예외결과물, 완료 기준, 예외 대응End 명시 + 예외 경로/루프 설계

6.3 현재 코드 기준 규칙

규칙군요약
라벨 규칙 RR-01~R-09 (Reject/Warning/Suggestion), 핵심: 금지동사/복수동작/목적어 누락
구조 규칙 SS-01~S-13, 핵심: 종료노드/고립/단절/조건라벨/루프/TO-BE 삭제연결
참고: First Guideline의 일부 항목(예: 주어 강제)은 현재 엔진에서 직접 강제하지 않으므로 작성 가이드로 운영한다.

7) 제품 기능/모드

7.1 구현 확인된 기능 (MD 전수검토 반영)

영역구현 상태핵심 내용
캔버스구현Start/End/Process/Decision/Subprocess, 엣지 라벨, 스윔레인(Y), Phase(X), undo/redo, JSON import/export
L7/구조 검증구현R-01~R-09, S-01~S-13 실시간/배치 검증, 이슈 포커싱, 원클릭 수정(R-04/R-05)
AI 코칭구현/api/chat, /api/review, /api/contextual-suggest, /api/first-shape-welcome
의도 분류구현knowledge / flow_action / coaching + 세부 의도(next/missing/decision/summary/review/general)
L345 참조 주입구현/api/chat, /api/review, /api/first-shape-welcome, /api/pdd-insights에 동적 컨텍스트 삽입
온보딩구현SetupModal, 온보딩 상태머신, 5단계 Product Tour, 빈 캔버스 가이드
모드구현AS-IS / TO-BE 2모드, TO-BE 카테고리 UI는 TO-BE 모드에서만 노출

7.2 모드별 사용 원칙

  • AS-IS: 현재 프로세스 문서화와 품질 진단(L7/구조 규칙 중심)
  • TO-BE: 카테고리 기반 재설계(as_is/digital_worker/ssc_transfer/delete_target/new_addition)
  • 예외: 병렬 게이트웨이(+)는 본 가이드에서 선반영하며, 상세 구현은 후속 반영 대상으로 관리
FEATURE_INVENTORY, USER_JOURNEY, GAP_ANALYSIS 등에서 계획/미구현으로 표시된 항목은 본 가이드 범위에서 제외했다.
