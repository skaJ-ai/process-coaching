# Process Coaching AI

L7 기반 HR 프로세스 설계를 위한 시각화 + 코칭 도구입니다.
ReactFlow 캔버스에서 프로세스를 그리고, FastAPI 백엔드의 AI 코칭/검증을 받아 단계 품질을 높이는 것이 목적입니다.

## 핵심 기능

- 플로우 차트 편집
  - 시작/종료/프로세스/판단/L6 프로세스 노드
  - 우클릭 컨텍스트 메뉴, 드래그 연결, 엣지 라벨 편집
- 스윔레인
  - 역할 구분선 ON/OFF
  - 레인 라벨 편집
  - 구분선 드래그/삭제
- L7 검증
  - 노드별 검증 및 배치 검증
  - 이슈 태그, 개선 제안, rewrite 추천
- AI 코칭
  - 플로우 리뷰, 채팅 Q&A, 맥락 제안(contextual suggest)
  - 제안 summary와 실제 라벨(labelSuggestion) 분리 적용
- 품질/운영 보조
  - 자동 저장, 내보내기/가져오기(JSON)
  - PDD 초안 생성

## 환경 변수

백엔드 시작 시 아래 파일들을 자동으로 읽습니다(위에서 아래 순서로 확인).

- `backend/.env`
- `backend/environment.txt`
- `.env` (프로젝트 루트)
- `environment.txt` (프로젝트 루트)

권장 방식:

1. `backend/environment.txt.example`를 복사해서 `backend/environment.txt` 생성
2. 주소/모델만 수정
3. 백엔드 재시작

- `LLM_BASE_URL` (default: `http://10.240.248.157:8533/v1`)
- `LLM_MODEL` (default: `Qwen3-Next`)
- `USE_MOCK`
  - `auto`: LLM 연결 확인 후 자동
  - `true`: 항상 mock 응답
  - `false`: 항상 LLM 사용 시도

예시:

```bash
# backend/environment.txt
LLM_BASE_URL=http://10.240.248.157:8533/v1
LLM_MODEL=Qwen3-Next
USE_MOCK=auto
```

참고: 셸에서 같은 키를 이미 설정했다면(`set LLM_BASE_URL=...`) 셸 값이 우선됩니다.

## L7 프로세스 작성 가이드라인

본 가이드라인은 HR 직원이 직접 업무를 **L7 수준으로 Decomposition**할 때, 모든 HR팀이 **동일한 품질**로 작성하도록 돕기 위한 기준이다.

### 원칙

* 필수 항목은 **100% 충족**해야 L7으로 인정
* 이 문장만 보고 다른 사람이 그대로 업무를 수행할 수 있어야 한다
* AS-IS 기준이며 개선/이상/자동화 관점은 포함하지 않는다

## 2. 고도화 근거 — v2 설계 이론적 배경

> **왜 이론적 근거가 필요한가?**
> 좋은 규칙은 "담당자가 만든 기준"이 아니라 **수천 개의 실제 프로세스 모델을 분석한 연구 결과**에서 나온다.
> v2는 "우리가 느끼기에 이상한 것"이 아니라, 글로벌 기업과 학계가 데이터로 검증한 기준을 HR 맥락에 맞게 번역한 것이다.
> 이를 통해 팀 내 논쟁을 줄이고("이 규칙이 왜 필요해?"), 외부 설명 가능성을 높일 수 있다.

### 2-1. bpmnlint — "프로세스 그림의 맞춤법 검사기"

**한 줄 요약**: SAP·BMW·Bosch 등 수십만 개 기업이 사용하는 프로세스 설계 도구 **Camunda**에 기본 내장된 자동 검증 엔진이다.

| 항목 | 내용 |
| :--- | :--- |
| **만든 곳** | Camunda (독일 BPM 전문 기업, 2021년 유니콘 달성) |
| **검증 규모** | 전 세계 10만+ 기업의 프로세스 모델에 적용 |
| **핵심 가치** | AI 없이 순수 규칙만으로 "연결이 끊겼다", "판단 노드에 조건이 없다"를 즉시 감지 |
| **우리 적용** | S-01~S-11 구조 규칙의 직접 출처 |

> **쉽게 말하면**: 한글 맞춤법 검사기가 문장 구조 오류를 잡아주듯, bpmnlint는 프로세스 플로우의 구조 오류를 잡아준다. "이 규칙이 왜 필요해?"라고 묻는다면, "Camunda가 10만 기업에 적용해서 효과를 검증했기 때문"이라고 답할 수 있다.

### 2-2. 7PMG — "프로세스가 복잡해지면 오류가 늘어난다는 데이터"

**한 줄 요약**: 2010년 Jan Mendling 교수(빈 경제대)가 실제 기업 프로세스 모델 604개를 분석해 도출한 **7가지 품질 원칙**이다. 이후 프로세스 관리 분야의 표준 참고 문헌이 됐다.

| 원칙 | 데이터가 말하는 것 | 우리 규칙 | 왜 중요한가 |
| :--- | :--- | :--- | :--- |
| G2 | 하나의 판단 노드에서 분기가 3개를 넘으면 **읽는 사람의 이해도가 급격히 떨어진다** | S-10 과다 분기 | 검토자가 플로우를 따라가지 못하면 오류를 놓친다 |
| G3 | 시작점이 2개 이상인 모델은 **실행 시 충돌이 발생할 가능성**이 높다 | S-09 다중 시작 | 프로세스의 시작 조건이 불명확해진다 |
| G7 | 노드가 50개를 넘으면 **모델 오류율이 50% 이상**으로 급등한다 | S-11 복잡도 | 단계가 너무 많으면 검토 자체가 불가능해진다 |

> **쉽게 말하면**: "복잡한 플로우일수록 오류가 많다"는 것을 직관이 아닌 실제 데이터로 증명한 연구다. 우리가 "50개 넘으면 나눠라"고 말할 때, 이는 개인 의견이 아니라 604개 기업 프로세스를 분석한 결과에 근거한 것이다.

### 2-3. BEF4LLM — "AI가 할 일과 규칙이 할 일을 구분하자"

**한 줄 요약**: 2025년 발표된 연구로, 프로세스 품질 검증에서 **AI(LLM)를 써야 하는 부분**과 **규칙 엔진으로 충분한 부분**을 체계적으로 구분하는 프레임워크다.

| 품질 차원 | 뜻 | 처리 방식 | 예시 |
| :--- | :--- | :--- | :--- |
| Validity (구조) | 연결이 완결됐는가 | 규칙 엔진 (S-규칙) | 고아 노드, 종료 노드 없음 |
| Syntactic (문법) | 라벨이 형식에 맞는가 | 규칙 엔진 (R-규칙) | 금지 동사, 목적어 누락 |
| Pragmatic (복잡도) | 사람이 이해할 수 있는가 | 규칙 엔진 (S-규칙) | 50개 초과, 4개 이상 분기 |
| Semantic (의미) | 문맥상 말이 되는가 | AI (LLM) | "이 단계가 맥락상 자연스러운가" |

> **핵심 결론**: 구조·형식·복잡도는 AI 없이 규칙만으로 70% 이상 커버할 수 있다. AI는 "의미가 맞는가"처럼 규칙으로 판단하기 어려운 영역에만 투입한다.
>
> **우리 적용**: AI 서버가 꺼져 있어도 S-규칙·R-규칙은 항상 동작한다. AI 판단이 "통과"여도 규칙 Reject는 유지된다.

---

### 2-4. Signavio — "구조 문제와 라벨 문제를 섞지 말자"

**한 줄 요약**: SAP가 인수한 글로벌 BPM 플랫폼 **Signavio**의 품질 피드백 UI 설계 원칙이다. 수만 개 기업의 UX 데이터를 기반으로, 사용자가 가장 빠르게 문제를 인식하는 방식을 연구했다.

| Signavio의 발견 | 우리 적용 |
| :--- | :--- |
| "이 플로우 뭔가 이상해" (구조 문제)와 "이 단어가 모호해" (라벨 문제)는 **인식 경로가 다르다** | 품질 대시보드에서 S-규칙(구조)과 R-규칙(라벨)을 별도 그룹으로 표시 |
| 두 가지를 섞어 보여주면 사용자가 **어디서부터 고쳐야 할지 모른다** | 구조 먼저 수정 → 라벨 다음 수정 순서를 자연스럽게 유도 |

> **쉽게 말하면**: "연결이 끊긴 문제"와 "동사가 모호한 문제"는 성격이 달라서 같이 보여주면 혼란스럽다. Signavio는 이것을 분리해서 보여줄 때 사용자 수정률이 올라간다는 것을 발견했고, 우리도 동일한 방식을 채택했다.

### 팀 토의 요청 사항

아래 항목은 현장 적용 전 팀 합의가 필요합니다.

1. **동사 분류 현장 검증** — 금지 9개 / 권장 13개 목록이 실제 HR 작성 패턴과 맞는가? 목록에서 빠진 동사가 있는가?
2. **보조한다 / 담당한다 처우** — v1에서는 금지였으나 v2에서 Warning으로 완화 제안. 지원한다(Reject)와의 일관성 측면에서 재검토 필요
3. **수행 위치 처리 방식** — v1 필수 항목이었으나 v2에서 메타데이터로 분리 제안. 라벨 간결성 vs. 수행 위치 명시 중 팀 우선순위 확인 필요
4. **구조 규칙 승격 여부** — S-01~S-11 중 Warning이 아닌 Reject으로 격상할 규칙이 있는가?
5. **플로우 종합 점수 가중치** — Structure 30% / Label 40% / Semantic 30% 비율이 적절한가?
