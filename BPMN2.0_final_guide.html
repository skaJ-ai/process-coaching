BPMN/L7 통합 최종 가이드

발행: Samsung DX AX & CI Lab / Process-Coaching 팀 · 버전: Final (2026-03-02) · 대상: 전사 HR 담당자

1. 프로젝트 목적과 통합 원칙

본 가이드는 기존 BPMN2.0_Complete_Guide.html, BPMN2.0_Complete_Guide copy.html, AX Lab 커스텀 원고를 통합한 최종본입니다. 핵심 목적은 단순 도식화가 아니라 L6/L7 기반의 데이터화를 통해 AX 전환 의사결정을 가능하게 만드는 것입니다.

핵심 질문: 이 업무가 끝났을 때 시스템에 어떤 디지털 흔적(Digital Trace)이 남는가?
  • 목표: HR 업무를 Outcome 중심으로 분해하고, Core/AI/SSC/Drop으로 구조적 분류
  • 원칙: BPMN 표준 우선, 툴 제약 후술, 구현 상태는 코드 기준으로만 반영
  • 산출물: AS-IS 완결 맵 + TO-BE 재설계 맵 + PDD/인사이트

2. BPMN 표준 대비 운영 프로파일

2.1 표준 근거

  • OMG BPMN 2.0.2 Conformance Class 기준 (Descriptive/Analytic/Common Executable)
  • 본 도구는 Descriptive 중심 실무 서브셋을 채택

2.2 현재 도구 프로파일

구분운영 정책
직접 지원Start/End, Process, Decision(XOR), Subprocess, Parallel(+), Sequence Flow, Edge Label, Swimlane, Phase
부분 지원Message Flow/고급 이벤트는 설명 레벨로만 사용
제외Inclusive/Event-based/Complex Gateway, Choreography/Conversation, 실행 엔진 의미론 상세
복잡한 실행 모델 표현보다, 현업 작성성과 품질 검증 가능성을 우선한다.

3. L6 Task 설계 원칙 (Outcome/Digital Trace)

모든 L6(Task)는 다음 Golden Standard를 충족해야 합니다.

  • 결과물 고립 원칙: "무엇을 했다"보다 "무엇이 남았다"로 정의
  • 단일 행위자 원칙: 한 담당자/시스템이 완결하는 최소 단위
  • 시간 단절 분리: 대기/회신/승인 대기 발생 시 새로운 L6로 분리
  • 주체 변경 분리: 책임 주체가 바뀌는 지점에서 캔버스 컷오프
  • 기계적 명칭 금지: 요청 접수, 유효성 검증 등 데이터 가치 낮은 포괄 표현 금지
BadGood (Outcome-based)
주재원 귀임 프로세스 진행귀임 완료 확인서 확정
채용 홍보 액팅채널별 공고 게시 로그 생성
사내 전화 상담S-calling 통화 이력 기록

4. 요소별 작성 가이드 (Samsung Ver.)

4.1 프로세스 노드 (L6/L7)

작성 형식: [산출물/대상] + [완료/확정/저장/발송/기록]

  • 한 셰이프는 한 동작만 표현
  • 복수 동작("검토하고 승인")은 분리
  • 시스템명은 라벨에서 분리하여 메타데이터로 기록

4.2 판단 노드 (Decision)

  • 형식 권장: ~여부, ~인가?, ~기한 이전인가?
  • 분기선 라벨(Yes/No 또는 조건식) 필수
  • 분기 1개면 Decision이 아니라 Process로 전환 검토

4.3 병렬 노드 (Parallel +)

  • Split/Join 쌍으로 설계
  • Split 출력 수와 Join 입력 수 정합성 확인
  • Join 없이 End로 바로 종료 금지

4.4 레인/페이즈

  • Swimlane(Y): 수행 주체(지원자/담당자/결재자 등)
  • Phase(X): 시간/단계 흐름 (3~5구간 권장)

6. L7/구조 규칙 체계 (코드 기준)

6.1 라벨 규칙 (R)

Rule핵심 의미
R-01/R-02라벨 길이 과소/과다
R-03a/R-03b금지 동사 reject / 구체화 권장 warning
R-04시스템명 라벨 혼입 감지 및 분리 권고
R-05복수 동작 reject (원클릭 분리 지원)
R-07타동사 목적어 누락 reject
R-08/R-09Decision 기준값/형식 점검

6.2 구조 규칙 (S)

Rule핵심 의미
S-01종료 노드 누락
S-02기본 라벨 미변경 노드
S-03/S-04고아 노드 / 나가는 연결 없음
S-05~S-08암묵 분기, 중복 연결, 무의미 Decision, 조건 라벨 누락
S-09~S-12다중 시작, 과다 분기, 복잡도 초과, 탈출 없는 루프
S-13TO-BE 삭제대상(delete_target) 연결 잔존
S-14/S-15병렬 Split/Join 짝 불일치 및 선행 Split 부재
원칙: L7 검증은 LLM 없이도 프론트 룰 엔진으로 즉시 동작한다.

7. AS-IS/TO-BE 작성 절차

7.1 AS-IS (현행 문서화)

  1. SetupModal에서 L4/L5/L6 컨텍스트 입력
  2. ModeSelector에서 AS-IS 분석 선택
  3. Start/End 배치 후 Happy Path 구성
  4. AI 인터뷰 시작 또는 채팅 제안으로 누락 단계 보강
  5. L7/구조 규칙 정리 + 메타데이터(systemName/duration/input/output) 채움
  6. QualityDashboard와 /api/review 기반 최종 검토

7.2 TO-BE (재설계/Cleansing)

AS-IS 정합 확보 후 TO-BE로 전환하여 노드를 분류합니다.

  • Core (유지) 사람 전문성 필수
  • AI (자동화) digital_worker 전환
  • SSC (이관) ssc_transfer 전환
  • Drop (제거) delete_target 지정
코드 기준 카테고리: as_is, digital_worker, ssc_transfer, delete_target, new_addition
