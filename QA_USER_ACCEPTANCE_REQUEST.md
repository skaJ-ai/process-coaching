# Process Coaching AI 사용자 관점 QA 요청서 (동료/현업 테스터용)

문서 버전: v1.0  
작성일: 2026-02-20

## 1) 목적
이번 QA의 목표는 자동화(E2E) 재확인이 아니라, 실제 사용자 관점에서 아래를 검증하는 것이다.

- 업무 담당자가 처음 진입했을 때 핵심 플로우를 스스로 완주할 수 있는가
- 화면/문구/상호작용이 혼란 없이 이해 가능한가
- 오류/예외 상황에서 복구 가능하고 신뢰 가능한가

## 2) 배경 (이미 자동화로 검증된 범위)
현재 Playwright 핵심 E2E는 아래를 이미 다룬다.

- F-01: 설정 모달(L4/L5/L6) 완료 후 메인 진입
- F-02: 노드 추가 및 라벨 입력
- F-03: L7 검증 신호(규칙/준수) 표시
- F-04: QualityDashboard 구조 이슈 액션(종료 노드 추가)
- F-05: 초안 저장/복원 흐름

따라서 이번 수동 QA는 기능 존재 여부보다 "실사용 완결성" 중심으로 수행한다.

## 3) 테스트 대상 기능 (현 구현 반영)
### 3.1 온보딩/진입
- SetupModal에서 L4/L5/L6 선택 후 드로잉 시작
- 초기 챗봇 환영 메시지 및 빠른 질문 버튼
- 온보딩/투어 표시 및 해제

### 3.2 캔버스 편집 핵심
- 우클릭 컨텍스트 메뉴로 셰이프 추가 (start/process/decision/subprocess/end)
- 노드 라벨 인라인 수정, 메타 정보 편집
- 엣지 연결/라벨링, 노드/엣지 삭제
- Undo/Redo
- 클립보드 복사/붙여넣기/선택 삭제

### 3.3 구조화 기능
- 역할 구분선(스윔레인) ON/OFF, 레인 추가(최대 4개 역할)
- 레인 라벨 변경 및 노드 자동 배치 반영

### 3.4 품질/검증
- L7 실시간/전체 검증, 경고/리젝트/통과 상태 표시
- QualityDashboard(R/S) 지표 및 "해당 노드 보기" 포커스 이동
- 완료하기 시 제출 게이트(종료 노드, 고아 노드, 미검증 노드 체크)

### 3.5 AI 코칭
- 전체 흐름 검토(`/api/review`)
- 채팅 Q&A(`/api/chat`)
- 맥락 제안(`/api/contextual-suggest`)
- 첫 노드 환영(`/api/first-shape-welcome`)
- (관리자) PDD 분석(`/api/analyze-pdd`, `/api/pdd-insights`)

### 3.6 저장/복구
- 중간저장(JSON 다운로드)
- 완료 제출(JSON 다운로드)
- 로컬 저장 데이터 복구
- Import/Export JSON 하위 호환(레거시 포맷 포함)

## 4) 테스트 방식
### 4.1 세션 구성
- 시나리오 기반 테스트: 정상/예외/경계 케이스
- 탐색형 테스트: 사용자 페르소나별 차터 수행 (아래 7장)

### 4.2 페르소나
- P1: HR 실무자(프로세스 모델링 경험 낮음)
- P2: HR 리드(검토/표준 준수 중시)
- P3: 운영자(복구/완료/산출물 확인 중시)

### 4.3 환경 매트릭스
- Desktop: Chrome 최신, Edge 최신
- Mobile View(브라우저 개발자도구): iPhone 14, Galaxy S22 폭 기준
- API 모드: LLM 연결 가능/불가(모크 fallback) 각각 1회 이상

## 5) 엔트리/종료 기준
### 5.1 엔트리 기준
- 프론트/백엔드 실행 가능
- 테스트 계정 또는 동일 조건의 초기 상태 준비
- 기존 자동화(E2E) 결과가 최신 커밋 기준으로 통과 또는 결과 공유됨

### 5.2 종료(승인) 기준
- P0 = 0건
- P1 = 0건 또는 승인권자 명시적 수용
- 핵심 사용자 시나리오 1~8 모두 통과
- "처음 사용자 관점" 개선 제안 최소 5건 수집

## 6) 핵심 사용자 시나리오 (필수)
각 시나리오는 "재현 단계 / 기대 결과 / 실제 결과 / 증빙"으로 기록한다.

1. 첫 진입 사용자가 5분 내 첫 프로세스 초안을 만든다.
- 기준: Setup -> 노드 3개 이상 추가 -> 연결 -> 라벨 입력 -> 중간저장 성공

2. 판단(Decision) 노드 사용 시 분기 라벨을 이해하고 올바르게 설정한다.
- 기준: 조건 라벨 누락 시 혼란 여부, 가이드 메시지 유효성

3. L7 전체 검증 후 어떤 노드를 먼저 고쳐야 하는지 즉시 파악한다.
- 기준: 대시보드 정보 해석 가능성, "해당 노드 보기" 동작 정확성

4. 오류 가능성이 있는 문장(모호 동사/복수 동작)을 사용했을 때 교정이 가능하다.
- 기준: 경고/리젝트 메시지의 이해도, rewrite 적용 후 개선 여부

5. 역할 구분선 없이 작성한 플로우를 레인 기반으로 재정리할 수 있다.
- 기준: 레인 ON/OFF, 레인 추가, 노드 재배치 인지 가능성

6. 채팅/전체 흐름 검토를 통해 다음 액션 제안을 실제로 반영할 수 있다.
- 기준: 제안의 구체성, 적용 동선의 마찰, 실패 시 대체 경로

7. 저장/복구/가져오기에서 데이터 유실 없이 재진입 가능하다.
- 기준: 다운로드 JSON 품질, 복구 다이얼로그, import 하위호환

8. 완료하기 시 품질 게이트 메시지를 이해하고 스스로 해결한다.
- 기준: 종료 노드 없음/고아 노드/미검증 경고 해소 후 완료 성공

## 7) 탐색형 테스트 차터 (사용자 체감 리스크 중심)
### Charter A: "처음 쓰는 사람의 막힘"
- 미션: 도움 없이 첫 초안 완성 가능 여부 확인
- 포커스: 용어 이해, 버튼 발견 가능성, 다음 행동의 명확성
- 타임박스: 45분

### Charter B: "복구 가능성"
- 미션: 실수/중단/오류 후 회복 가능 여부 확인
- 포커스: 되돌리기, 복구, 재시도, 메시지 품질
- 타임박스: 30분

### Charter C: "코칭 신뢰성"
- 미션: AI 제안이 실제 업무 모델링에 도움이 되는지 검증
- 포커스: 모호/과도/부정확 제안, 적용시 부작용
- 타임박스: 30분

## 8) 심각도/우선순위 기준
- P0 (Release Blocker): 데이터 유실, 저장/복구 실패, 완료 불가, 주요 흐름 진입 불가
- P1 (Critical): 핵심 작업 성공률을 크게 떨어뜨리는 UX/로직 결함
- P2 (Major): 우회 가능하나 품질 저하가 큰 결함(오해 유발 문구 포함)
- P3 (Minor): 경미한 UI/문구/일관성 문제

참고: 심각도는 사용자 영향도 기준으로 판단하고, 우선순위는 릴리즈 일정/비즈니스 영향으로 별도 조정한다.

- 실행 시나리오 수 / 통과 / 실패 / 보류
- P0~P3 결함 카운트

### 10.2 결론

## 동료에게 보내는 요청 메시지 (복붙용)
안녕하세요. 이번 QA는 자동화 재검증이 아니라 실제 사용자 관점 검증이 목적입니다.  
첨부한 `QA_USER_ACCEPTANCE_REQUEST.md` 기준으로 시나리오 1~8 + 탐색 차터 A/B/C를 수행 부탁드립니다.  
특히 "처음 사용자 막힘", "오류 후 복구", "코칭 제안의 실효성"을 중점으로 확인해 주세요.  
버그는 템플릿 필수 항목으로 작성 부탁드립니다.
