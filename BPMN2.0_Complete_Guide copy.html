Process-Coaching BPMN/L7 가이드북

버전: 2026-03-02 · 목적: HR AX 프로세스 재설계 및 플랫폼화

문서 원칙: 1) BPMN 글로벌 표준은 가공 없이 우선 제시, 2) 우리 툴 해석/제약은 후술, 3) 미구현 기능은 제외(병렬 게이트웨이는 가이드 선반영 예외).

1) 프로젝트 목적 (AX 전환)

본 프로젝트의 궁극적인 목적은 단순히 현재 업무(AS-IS)를 예쁘게 그리는 문서화 작업이 아닙니다. HR 업무 중 '고부가가치 업무(Core)'만 남기고, 나머지는 플랫폼(AI/SSC)에 넘겨 일하는 방식을 혁신(AX)하는 것입니다.

  • 목적: 업무를 기계(AI/SSC)가 소화할 수 있는 단위(L6)로 쪼개어, 유지/자동화/이관/삭제 대상을 구조적으로 솎아낸다(Cleansing).
  • 산출물: 이해 가능한 프로세스 모델 + 4대 속성이 라벨링된 TO-BE 모델
  • 운영 원칙: 현업이 직접 '진짜 L6'를 쪼갤 수 있어야 하며, 툴의 규칙 엔진으로 품질을 일관되게 통제한다.

2) 설계 근거 (BPMN 표준)

2.1 글로벌 표준 기준

  • 기준 문서: OMG BPMN 2.0.2 (Conformance/Process Modeling)
  • Conformance는 모델링 계열과 실행 계열을 포괄
  • Process Modeling은 Descriptive / Analytic / Common Executable 하위 클래스로 구분

2.2 Descriptive Sub-Class 핵심

분류대표 요소 (OMG Table 2.1 기반)
흐름sequenceFlow, messageFlow
게이트웨이exclusiveGateway, parallelGateway
활동task, userTask, serviceTask, subProcess
이벤트startEvent, endEvent
구조participant, lane, DataObject, TextAnnotation

3) 우리 툴 모델링 프로파일

우리는 Descriptive 철학을 차용하되, 제품 목적에 맞게 요소를 선별한 실무 프로파일로 운영한다.

4) 요소별 작성 가이드

4.1 시작/종료 노드

  • Start는 진입점, End는 완료 조건을 명확히 닫는다.
  • Start 다중 사용은 의도적 구조일 때만 허용, End는 최소 1개 필수.

4.2 프로세스 노드 (Task)

  • 동사 + 목적어로 작성: 예) 결재 요청서를 검토한다
  • 한 셰이프=한 동작 원칙 준수
금지 예: "검토하고 승인한다"(복수 동작), "업무를 처리한다"(모호 동사)

5) '진짜 L6' 정의 및 빈 캔버스 작성 절차

⚠️ 스파게티 맵 경고: 사람과 기계의 역할, 며칠간의 흐름을 하나의 캔버스에 모두 구겨 넣으면 AI가 해석할 수 없는 '스파게티 맵'이 됩니다. 캔버스를 열기 전 반드시 아래 기준에 따라 업무(L6)를 먼저 쪼개십시오.

5.1 스파게티 맵 탈출을 위한 L6 컷오프(Cut-off) 3대 원칙

다음 세 가지 중 하나라도 발생하면, 선을 긋고 새로운 L6 캔버스로 분리(Cut)해야 합니다.

  • ① 결과물 고립 원칙 (Outcome): 이 덩어리만 뚝 떼어 남에게 시켰을 때, 완결된 '독립적 결과물(Output)'이 나오는가? 여러 결과물이 섞여 있다면 자른다.
  • ② 시간 단절 (One Sitting): 담당자가 자리에 앉아 중간에 타인의 회신이나 승인을 기다리지 않고 한 번에 끝낼 수 있는가? 대기(Wait)가 발생하면 자른다.
  • ③ 주체 변경 (Handoff): 업무의 주도권이나 책임 소재가 완전히 타 부서나 타 시스템으로 넘어가는가? 넘어가면 자른다.

5.2 캔버스 작성 순서 (Top-down)

  1. 범위 고정: 위 3대 원칙을 통과한 '진짜 L6' 단위 하나만 타겟으로 잡아 Start/End를 배치한다.
  2. 구획 정의: 역할 구분선(Y) 및 Phase 구분선(X)을 그어 캔버스의 논리적 구조를 먼저 잡는다.
  3. Happy Path 연결: 화살표 없이 태스크(L7) 후보를 먼저 배치하고, 가장 정상적인 흐름을 일직선으로 연결한다.
  4. 예외/분기 처리: Decision/Parallel을 필요한 지점에 추가하되, 반드시 End로 향하는 탈출 경로를 확인한다.

7) 제품 기능 및 4대 속성(Cleansing) 태깅

7.1 구현 확인된 기능

영역구현 상태핵심 내용
캔버스구현Start/End/Process/Decision/Subprocess, 엣지 라벨, 스윔레인(Y), Phase(X)
L7/구조 검증구현R-01~R-09, S-01~S-13 실시간/배치 검증, 이슈 포커싱, 원클릭 수정
AI 코칭구현/api/chat, /api/review, /api/contextual-suggest
모드구현AS-IS / TO-BE 2모드 전환

7.2 모드별 사용 원칙 및 4대 속성 솎아내기 (TO-BE)

AS-IS 모드에서 정합성 높은 L6 캔버스가 완성되었다면, TO-BE 모드로 전환하여 '플랫폼 이관 대상 솎아내기(Cleansing)'를 진행합니다. 각 셰이프(L7) 또는 덩어리에 다음 4가지 속성 중 하나를 반드시 태깅하십시오.

  • Core (유지) 내가 남길 일: 고도의 예외 상황 판단, 기획, 전략적 의사결정 등 사람의 전문성이 필수적인 업무
  • AI (자동화) 디지털 워커: 규칙 기반 데이터 대조, 일정 핑퐁 조율, 초안 작성, 시스템 자동 입력 등 AI/RPA 처리 가능 업무
  • SSC (이관) 공유 서비스: 단순 반복, 정형화된 CS 응대, 서류 단순 취합 등 타 조직으로 떼어낼 수 있는 업무
  • Drop (제거) 폐기 대상: 불필요한 관행적 보고, 중복 데이터 입력, 무의미한 중간 결재
결과: TO-BE 맵에서는 AI와 SSC로 분류된 노드들이 '플랫폼 블랙박스'로 묶이고, Core 업무만 담당자의 화면에 남게 됩니다.

8) 현업용 실전 체크리스트 (Self-Check)

캔버스를 닫기 전, 스스로 아래 질문에 모두 'Yes'라고 답할 수 있는지 점검하십시오.
  • Q1. 노드 개수: 캔버스 하나에 L7 노드(태스크, 판단 등)가 15개를 초과하지 않는가? (초과 시 너무 크게 잡은 것임)
  • Q2. 결과물 확인: 이 캔버스가 끝났을 때 도출되는 '명확한 1개의 결과물(Output)'을 말할 수 있는가?
  • Q3. 시간의 연속성: 맵 중간에 타인의 회신이나 날짜를 기다리며 업무가 완전히 멈추는 구간이 없는가?
  • Q4. 속성 분리: 캔버스 내의 업무 중 '기계(AI)가 할 일'과 '사람이 판단할 일(Core)'이 시각적으로 완벽히 분리 및 태깅되었는가?
